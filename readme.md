# Документация шаблона сборки (Vite + PostCSS + LightningCSS + EJS)

Данный проект представляет собой оптимизированный стек для современной адаптивной верстки. Сборщик базируется на Vite, шаблонизаторе EJS и продвинутой архитектуре PostCSS с использованием LightningCSS для минификации.

---

## 📋 Краткий справочник команд CLI

| Команда                   | Описание                                                                                                                                                                     |
| :------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `bun dev`                      | Запуск локального сервера разработки Vite со встроенным HMR.                                                                            |
| `bun build`                    | Компиляция проекта под продакшен (сборка HTML, оптимизация CSS, кэш-бастинг).                                                |
| `bun run convert_images`       | Нарезка адаптивной графики по сетке размеров и современным форматам из `/public/src/img/`.                            |
| `bun run convert_images:clean` | Интерактивная очистка задействованных папок размеров с последующим запуском новой конвертации.   |
| `bun run clean_images`         | Полная генеральная уборка. Автоматический поиск и ручная очистка всех папок результатов вида `*w`. |
| `bun run sort:css`             | Автоматическое форматирование и логическая сортировка CSS-свойств внутри всех селекторов.                |

## 🚀 Быстрый старт

В качестве основного окружения выполнения и менеджера пакетов в проекте используется **Bun** — современная, ультрабыстрая альтернатива Node.js, npm и Yarn.

Bun берет на себя функции пакетного менеджера, рантайма и транспайлера. На базе Bun данная сборка показала наилучшие результаты: скорость установки зависимостей увеличилась в разы, а запуск локального сервера разработки и компиляция стилей происходят практически мгновенно.

> Проект также полностью совместим с Node.js + npm / yarn — все скрипты написаны универсально.

### Команды управления:

1. Установка зависимостей:

```bash
bun install
```

2. Запуск сервера разработки:

```bash
bun dev
```

3. Сборка продакшена:

```bash
bun build
```

## 🛠 Конфигурация окружения и деплой статики

Папки статических ресурсов (`public/img/` и `public/fonts/`) исключены из процесса сборки Vite в режиме production — они не копируются в папку `dist` автоматически. Это сделано для того, чтобы не нагружать сборщик гигабайтами графики. Медиа-файлы и шрифты должны загружаться на продакшен-сервер или CDN отдельно.

Управление путями к картинкам на продакшене осуществляется через файл `.env.production` в корне проекта:

```env
VITE_PROD_IMG_PREFIX=//your-cdn.com/img/
```

Если файл `.env.production` отсутствует или переменная не задана, сборщик автоматически применит безопасный относительный фолбек: `img/`.

## 📐 Работа с HTML и шаблонами (EJS)

### 📦 Инклуды компонентов

Благодаря настройке `ejsOptions.root`, пути во всех инклудах указываются строго от корня проекта с ведущим слэшем `/`. Никаких путаных относительных путей вида `../../`.

*Внутри `index.html` в корне:*

```html
<%- include('/src/blocks/trends/trends.html') %>
```

*Внутри `/src/blocks/trends/trends.html` (подключение соседа):*

```html
<%- include('/src/blocks/signal/signal.html') %>
```

### 🖼 Пути к картинкам в HTML

Поскольку локальный сервер разработки смотрит напрямую в папку `public/img/` как в корень статики, в коде вы пишете только имя файла без префиксов папок.

При сборке плагин автоматически трансформирует пути: добавляет префикс из `.env.production` и короткий хэш версии для сброса кэша браузера.

#### Вариант 1: Обычная картинка

*В коде:*

```html
<img src="IG3.webp" alt="Картинка">
```

*На продакшене:*

```html
<img src="//your-cdn.com/img/IG3.webp?lzz8x12" alt="Картинка">
```

#### Вариант 2: Картинка с ручным суффиксом

Если у картинки уже указан знак вопроса, авто-версионирование игнорируется — сохраняется ваш суффикс, но добавляется хост CDN.

*В коде:*

```html
<img src="photo.jpg?1" alt="Картинка">
```

*На продакшене:*

```html
<img src="//your-cdn.com/img/photo.jpg?1" alt="Картинка">
```

#### Вариант 3: Современный адаптивный `<picture>`

Плагин корректно обрабатывает атрибуты `srcset`.

*В коде:*

```html
<picture>
  <source srcset="hero.avif" type="image/avif">
  <img src="hero.jpg" alt="Главная">
</picture>
```

*На продакшене:*

```html
<picture>
  <source srcset="//your-cdn.com/img/hero.avif?lzz8x12" type="image/avif">
  <img src="//your-cdn.com/img/hero.jpg?lzz8x12" alt="Главная">
</picture>
```

## 🎨 Работа со стилями (CSS / PostCSS)

Стили могут располагаться в папке `/src/` с любым уровнем вложенности.

### 🔌 Подключение стилей (импорты)

Используется единая точка входа, в которой слои интерфейса подключаются последовательно через `@import` с абсолютными путями от корня проекта:

```css
@import '/src/styles/reset.css';
@import '/src/styles/fonts.css';
@import '/src/styles/common.css';
@import '/src/styles/ui.css';
@import '/src/styles/layout.css';
```

### 🅰 Подключение шрифтов

Шрифты физически располагаются в папке `public/fonts/`. Пути в CSS указываются относительно корня статического сервера:

```css
@font-face {
  font-family: 'Aptos';
  src: url('fonts/Aptos.woff2') format('woff2');
  font-weight: 400;
  font-style: normal;
  font-display: swap;
}
```

### 📉 Вложенность селекторов (postcss-nested)

Интегрирован плагин `postcss-nested`, позволяющий писать модульный CSS с вложенностью в стиле SASS. Идеально подходит для методологии БЭМ с использованием амперсанда (`&`):

```css
.card {
  padding: 20px;
  background: #fff;

  &__title {
    font-size: 18px;
    color: #000;
  }

  &--featured {
    border: 2px solid red;
  }

  &:hover {
    box-shadow: 0 4px 10px rgba(0, 0, 0, 0.1);
  }
}
```

### 🗃 Сортировка CSS-свойств (Prettier)

Для автоматической сортировки CSS-свойств внутри селекторов используется Prettier. Запускается вручную перед коммитом или продакшен-сборкой:

```bash
bun run sort:css
```

### 📐 Медиа-миксин `@include media`

Для управления брейкпоинтами используется кастомный PostCSS-плагин `media`. Вместо громоздких нативных `@media`-конструкций используется лаконичный синтаксис с операторами диапазона.

#### Доступные именованные брейкпоинты:

| Алиас     | Значение |
| :------------- | :--------------- |
| `max_mobile` | 640px            |
| `desk`       | 1198px           |
| `desk_l`     | 1440px           |

#### Операторы диапазона:

| Оператор | Смысл                       |
| :--------------- | :------------------------------- |
| `>=`           | от (включительно)  |
| `<=`           | до (включительно)* |
| `>`            | строго больше        |
| `<`            | строго меньше        |

*\* При компиляции в классический синтаксис автоматически вычитается `0.02px` для точного стыка брейкпоинтов.*

#### Синтаксис:

В качестве значений принимаются именованные алиасы или конкретные значения в `px`, `rem`, `em`:

```css
/* Одно условие или диапазон — плоская запись */
@include media(">=max_mobile", "<=desk") { ... }

/* Несколько независимых групп через запятую */
@include media((">=desk", "<=desk_l"), (">=1366px", "screen", "portrait")) { ... }
```

#### Дополнительные алиасы медиа-условий:

| Алиас         | CSS-условие                   |
| :----------------- | :----------------------------------- |
| `portrait`       | `(orientation: portrait)`          |
| `landscape`      | `(orientation: landscape)`         |
| `dark`           | `(prefers-color-scheme: dark)`     |
| `light`          | `(prefers-color-scheme: light)`    |
| `hover`          | `(hover: hover)`                   |
| `no-hover`       | `(hover: none)`                    |
| `reduced-motion` | `(prefers-reduced-motion: reduce)` |

#### Примеры использования:

```css
.sidebar {
  width: 100%;

  @include media(">=desk", "<=desk_l") {
    width: 300px;
    padding: 20px;
  }

  @include media("<desk") {
    padding: 10px;
  }
}
```

#### Скомпилированный результат (классический режим):

```css
.sidebar { width: 100%; }

@media only screen and (min-width: 1198px) and (max-width: 1439.98px) {
  .sidebar { width: 300px; padding: 20px; }
}

@media only screen and (max-width: 1197.98px) {
  .sidebar { padding: 10px; }
}
```

#### Автоматический выбор синтаксиса (Level 4 vs Classic):

При запуске сборщик анализирует `browserslist` проекта и автоматически выбирает синтаксис медиа-запросов:

- Если все браузеры в списке поддерживают Media Queries Level 4 — генерируется современный синтаксис `(width >= N)` без epsilon-поправок (браузер обрабатывает границы сам).
- Если в списке есть браузеры без поддержки Level 4 — генерируется классический `min-width` / `max-width` с автоматическим вычетом `0.02px`.

В консоли при старте отображается активный режим:

```
[media-dsl] Mode: Level 4 — (width >= N) / (width <= N)
```

или

```
[media-dsl] Mode: Classic — (min-width: N) / (max-width: N±epsilon)
```

### 🌊 Резиновые свойства (`fluid_prop`)

Кастомный PostCSS-плагин `fluid_prop()` рассчитывает линейную зависимость размера любого CSS-свойства на основе четырёх параметров: минимального/максимального экрана и минимального/максимального значения свойства.

На выходе генерируется нативная конструкция из `max()` и `min()`, предотвращающая выход за указанные лимиты.

#### Синтаксис:

```css
property: fluid_prop(minScreen, maxScreen, minSize, maxSize);
```

#### Пример использования:

```css
.hero__title {
  font-size: fluid_prop(1025px, 1920px, 24px, 48px);
  padding-top: fluid_prop(320px, 1440px, 20px, 60px);
}
```

#### Скомпилированный результат:

```css
.hero__title {
  font-size: max(24px, min(calc(-3.531px + 2.682vw), 48px));
  padding-top: max(20px, min(calc(8.571px + 3.571vw), 60px));
}
```

## 🖼️ Оптимизация и конвертация изображений (`convert.js`)

CLI-скрипт `convert.js` сканирует исходники в `/public/src/img/` и нарезает папки с нужными размерами и форматами для тегов `<picture>`.

### Технологический стек:

- **`sharp`** — сверхбыстрый модуль для обработки графики, значительно быстрее ImageMagick.
- **`commander`** — управление параметрами и флагами командной строки.
- **`glob`** — поиск файлов по маскам с поддержкой любой глубины вложенности.

### Безопасность и архитектура:

**Жёсткая изоляция путей.** Скрипт блокирует абсолютные пути и попытки выйти за пределы проекта через `../`. Все действия происходят строго внутри:

- Исходные изображения: `./public/src/img/`
- Сгенерированные копии: `./public/img/`

**Флаг `--clean`.** При очистке скрипт никогда не удаляет директории вслепую. Он поштучно проверяет расширение каждого файла и удаляет исключительно графические файлы (`.jpg`, `.jpeg`, `.png`, `.webp`, `.avif`). Посторонние файлы сохраняются, а папка пропускается с предупреждением. Перед очисткой всегда запрашивается интерактивное подтверждение `(y/n)`.

### Качество сжатия:

Дополнительная оптимизация после работы скрипта не требуется — он выдаёт готовый промышленный уровень сжатия:

- **WebP и AVIF:** задействованы максимальные параметры кодеков с глубоким анализом проходов (`effort: 4`).
- **JPG:** интегрирован алгоритм **mozjpeg** от Mozilla с прогрессивным сканированием и квантованием методом Trellis. Файлы с качеством `82` весят на ~25% меньше аналогов из Photoshop.
