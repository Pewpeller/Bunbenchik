# Документация шаблона сборки (Vite + PostCSS + LightningCss + EJS)

Данный проект представляет собой оптимизированный стек для современной адаптивной верстки. Сборщик базируется на Vite, шаблонизаторе EJS и продвинутой архитектуре PostCSS (с использованием LightningCSS для минимизации).

---

## 📋 Краткий справочник команд CLI

| Команда                   | Описание                                                                                                                                                                                        |
| :------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `bun dev`                      | Запуск локального сервера разработки Vite со встроенным HMR.                                                                                               |
| `bun build`                    | Компиляция проекта под продакшен (сборка HTML, оптимизация CSS, кэш-бастинг).                                                                   |
| `bun run convert_images`       | Нарезка адаптивной графики по сетке размеров и современным форматам из `/public/src/img/`.                                               |
| `bun run convert_images:clean` | Интерактивная очистка задействованных папок размеров с последующим запуском новой конвертации.                      |
| `bun run clean_images`         | Полная генеральная уборка. Автоматический поиск и ручная очистка абсолютно всех папок результатов вида `*w`. |
| `bun run sort:css`             | Автоматическое форматирование и логическая сортировка CSS-свойств внутри всех селекторов.                                   |

## 🚀 Быстрый старт

В качестве основного окружения выполнения и менеджера пакетов в проекте используется **Bun** — современная, ультрабыстрая альтернатива Node.js, npm и Yarn.

Bun берет на себя функции пакетного менеджера, рантайма и транспайлера. На базе Bun данная сборка показала наилучшие результаты: скорость установки зависимостей увеличилась в разы, а запуск локального сервера разработки и компиляция стилей происходят практически мгновенно и со стопроцентной стабильностью.

### Команды управления:

1. Установка зависимостей:

```bash
  bun install
```

2. Запуск сервера разработки (dev):

```bash
  bun dev
```

3. Сборка продакшена (build):


## 🛠 Конфигурация окружения и деплой статики

Папки статических ресурсов (`public/img/` и `public/fonts/`) исключены из процесса сборки Vite в режиме production (они не копируются в папку `dist` автоматически). Это сделано для того, чтобы не нагружать сборщик гигабайтами графики. Медиа-файлы и шрифты должны загружаться на продакшен-сервер или CDN отдельно.

Управление путями к картинкам на продакшене осуществляется через файл `.env.production` в корне проекта:

```env
VITE_PROD_IMG_PREFIX=//your-cdn.com/img/
```

Если файл `.env.production` отсутствует или закомментирован, сборщик автоматически применит безопасный относительный фолбек: `img/`.

## 📐 Работа с HTML и шаблонами (EJS)

### 📦 Инклуды компонентов

Благодаря настройке `ejsOptions.root`, пути во всех инклудах (как в корневых файлах, так и во вложенных блоках) указываются строго от корня проекта с ведущим слэшем `/`. Больше никаких путаных относительных путей вида `../../`.

*Внутри index.html в корне:*

```html
<%- include('/src/blocks/trends/trends.html') %>
```

*Внутри /src/blocks/trends/trends.html (подключение соседа):*

```html
<%- include('/src/blocks/signal/signal.html') %>
```

### 🖼 Пути к картинкам в HTML

Поскольку локальный сервер разработки смотрит напрямую в папку `public/img/` как в корень статики, в коде вы пишете только имя файла (без префиксов папок).

При сборке плагин автоматически трансформирует пути, добавляя префикс из `.env.production` и короткий хэш версии для сброса кэша браузера (кэш-бастинг).

#### Вариант 1: Обычная картинка

*В коде:*

```html
<img src="IG3.webp" alt="Картинка">
```

*На продакшене:*

```html
<img src="//your-cdn.com/img/IG3.webp?lzz8x12" alt="Картинка">
```

#### Вариант 2: Картинка с ручным суффиксом

Если у картинки уже указан знак вопроса, авто-версионирование игнорируется, сохраняя ваш суффикс, но добавляя хост CDN.

*В коде:*

```html
<img src="photo.jpg?1" alt="Картинка">
```

*На продакшене:*

```html
<img src="//your-cdn.com/img/photo.jpg?1" alt="Картинка">
```

#### Вариант 3: Современный адаптивный &lt;picture&gt;

Применяется аналогично, плагин корректно обрабатывает атрибуты `srcset`.

*В коде:*

```html
<picture>
  <source srcset="hero.avif" type="image/avif">
  <img src="hero.jpg" alt="Главная">
</picture>
```

*На продакшене:*

```html
<picture>
  <source srcset="//your-cdn.com/img/hero.avif?lzz8x12" type="image/avif">
  <img src="//your-cdn.com/img/hero.jpg?lzz8x12" alt="Главная">
</picture>
```

## 🎨 Работа со стилями (CSS / PostCSS)

Стили могут располагаться в папке `/src/` с любым уровнем вложенности (например, `/src/styles/reset.css` или `/src/blocks/stage/stage.css`).

### 🔌 Подключение стилей на странице (Импорты)

Сборка использует единую точку входа (например, главный CSS-файл), в котором слои интерфейса и системные стили подключаются последовательно с использованием абсолютных путей от корня проекта через директиву `@import`. Это гарантирует строгий порядок каскада и исключает путаницу с путями.

```css
@import '/src/styles/reset.css';
@import '/src/styles/fonts.css';
@import '/src/styles/common.css';
@import '/src/styles/ui.css';
@import '/src/styles/layout.css';
@import '/src/styles/vicons.css';
```

### 🅰 Подключение шрифтов

Шрифты физически располагаются в папке `public/fonts/`. В CSS-файлах пути к ним указываются относительно корня статического сервера. Поскольку папка fonts является подкаталогом статики, путь всегда начинается с `fonts/`.

Пример подключения в файле `/src/styles/fonts.css`:

```css
@font-face {
  font-family: 'Aptos';
  src: url('fonts/Aptos.woff2') format('woff2');
  font-weight: 400;
  font-style: normal;
  font-display: swap;
}
```

### 📉 Вложенность селекторов (postcss-nested)

В сборку интегрирован плагин `postcss-nested`, позволяющий писать модульный CSS с использованием вложенности в стиле SASS/LESS. Это идеально подходит для методологии BEM (БЭМ) с использованием амперсанда (`&`).

Пример использования:

```css
.card {
  padding: 20px;
  background: #fff;

  /* Элемент БЭМ (.card__title) */
  &__title {
    font-size: 18px;
    color: #000;
  }

  /* Модификатор БЭМ (.card--featured) */
  &--featured {
    border: 2px solid red;
  }

  /* Стили при ховере */
  &:hover {
    box-shadow: 0 4px 10px rgba(0, 0, 0, 0.1);
  }
}
```

### 🗃 Сортировка CSS свойств (prettier)

Для поддержания идеального порядка и стандартизации кода в проекте используется плагин Prettier для автоматической сортировки CSS-свойств внутри селекторов.

Сортировка запускается строго вручную с помощью команды:

```bash
bun run sort:css
```

Рекомендуется выполнять данную команду перед компиляцией продакшен-версии проекта (`bun build`), чтобы исходный код в репозитории оставался чистым, структурированным и легко читаемым для всей команды.

### 📐 Адаптивный миксин @mixin media

Для управления брейкпоинтами используется кастомный PostCSS-миксин `media`. Больше нет необходимости писать громоздкие нативные конструкции `@media`. Миксин автоматически преобразует именованные точки в медиа-запросы вида `only screen and (...)`.

Доступные ключевые точки (заданы в конфигурации Vite):

* `desk` (1025px)
* `desk_l` (1441px)
* `desk_l_gap` (1521px)
* `desk_xl` (1920px)

#### Вариант написания и брейкпоинты:

Миксин поддерживает гибкий синтаксис на базе ключевых слов `from` (нижняя граница) и `to` (верхняя граница):

* `@mixin media from desk { ... }` — Стили активны начиная с ширины экрана 1025px и выше.
* `@mixin media to desk_l { ... }` — Стили активны на экранах до 1440.98px (автоматический технический вычет 0.02px для стыка).
* `@mixin media from desk to desk_l { ... }` — Стили активны строго в диапазоне от 1025px до 1440.98px одновременно.

#### Пример использования в исходном CSS:

```css
.sidebar {
  width: 100%;

  /* Ограничение области видимости от десктопа до большого десктопа */
  @mixin media from desk to desk_l {
    width: 300px;
    padding: 20px;
  }

  /* Применение стилей только до базового десктопа */
  @mixin media to desk {
    padding: 10px;
  }
}
```

#### Что мы получим на выходе (скомпилированный CSS):

```css
.sidebar {
  width: 100%;
}
@media only screen and (min-width: 1025px) and (max-width: 1440.98px) {
  .sidebar {
    width: 300px;
    padding: 20px;
  }
}
@media only screen and (max-width: 1024.98px) {
  .sidebar {
    padding: 10px;
  }
}
```

### 🌊 Резиновые свойства (fluid_prop)

Кастомный PostCSS-плагин `fluid_prop()` автоматически рассчитывает линейную зависимость размера любого CSS-свойства (шрифты, отступы, высоты) на основе четырех строго обязательных параметров: минимального/максимального экрана и минимального/максимального значения свойства.

На выходе он генерирует нативную безопасную зажим-конструкцию из функций `max()` и `min()`, предотвращая неконтролируемый выход за рамки указанных лимитов.

#### Синтаксис:

```css
property: fluid_prop(minScreen, maxScreen, minSize, maxSize);
```

#### Пример использования:

```css
.hero__title {
  /* Значение будет плавно рассчитываться на экранах от 1025px до 1920px в пределах от 24px до 48px */
  font-size: fluid_prop(1025px, 1920px, 24px, 48px);

  /* Резиновый внутренний отступ для другого диапазона */
  padding-top: fluid_prop(320px, 1440px, 20px, 60px);
}
```

#### Что мы получим на выходе (скомпилированный CSS):

Обработчик высчитает точный наклон и смещение, заменив функцию чистым математическим выражением:

```css
.hero__title {
  font-size: max(24px, min(calc(-3.531px + 2.682vw), 48px));
  padding-top: max(20px, min(calc(8.571px + 3.571vw), 60px));
}
```

## 🖼️ Оптимизация и автоматическая конвертация изображений (`convert.js`)

Для автоматизации подготовки адаптивной графики в корень проекта интегрирован служебный CLI-скрипт `convert.js`. Он сканирует исходники в корневой папке проекта /public/src/img и поштучно нарезает папки с нужными размерами и форматами для тегов `<picture>`.

### Технологический стек:

* **`sharp`** — Сверхбыстрый и производительный модуль ядра Node.js для работы с графикой. Он обрабатывает изображения в разы быстрее устаревших утилит вроде ImageMagick.
* **`commander`** — Мощная библиотека для гибкого управления параметрами и флагами командной строки.
* **`glob`** — Поиск файлов по маскам, обеспечивающий сканирование папок любой глубины вложенности.

### Безопасность и архитектура:

1. **Жесткая изоляция путей:** Скрипт блокирует ввод абсолютных путей или попытки выйти из проекта через `../`. Все действия происходят строго внутри фиксированных папок.

* Исходные изображения: `./public/src/img/`
* Сгенерированные адаптивные копии: `./public/img/`

2. **Флаг `--clean`:** При очистке результатов скрипт **никогда не удаляет директории вслепую**. Он заходит в каждую папку размера (например, `1200w/`), поштучно проверяет расширение каждого файла и удаляет исключительно графические файлы (`.jpg`, `.jpeg`, `.png`, `.webp`, `.avif`). Если в папку случайно попал сторонний файл (скрипт, документ, лог), утилита **сохранит его**, пропустит папку и выдаст предупреждение. Перед выполнением очистки скрипт всегда запрашивает интерактивное подтверждение `(y/n)`.

### Стоит ли дополнительно оптимизировать картинки после работы скрипта?

**Нет, это абсолютно бессмысленно.** Скрипт выдает готовый промышленный уровень сжатия, оптимизированный под стандарты Google PageSpeed. Повторный прогон через сторонние CLI-компрессоры не уменьшит вес файлов, но существенно увеличит время сборки проекта.

* **Для WebP и AVIF:** Внутри `sharp` задействованы максимальные параметры кодеков с глубоким анализом проходов (`effort: 4`), обеспечивающие экстремальное сжатие без появления мыла и шумов.
* **Для JPG (JPEG):** В скрипт интегрирован алгоритм **`mozjpeg: true`** от Mozilla. Он автоматически переводит файлы в формат прогрессивного сканирования (картинка весит на 10% меньше и загружается на сайте плавно от размытого к четкому) и использует квантование таблиц методом Trellis. На выходе файлы с качеством `82` весят на четверть меньше стандартных аналогов из Photoshop.

### Команды автоматизации (запуск через Bun или npm):

* `bun run convert_images` — Стандартный запуск. Нарезает изображения по сетке размеров под нужный формат.
* `bun run convert_images:clean` — Сначала интерактивно запрашивает подтверждение, очищает от старой графики только те папки размеров, которые указаны в текущей команде, и сразу генерирует новые.
* `bun run clean_images` (или `node convert.js --clean`) — Автономный режим дворника. Скрипт сам сканирует папку `converted`, находит вообще все существующие каталоги размеров вида `*w`, запрашивает подтверждение, поштучно вычищает графику и полностью удаляет опустевшие папки, освобождая дисковое пространство.
